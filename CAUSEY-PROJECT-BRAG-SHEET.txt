  ├─ public.waitlist
  └─ public.api_rate_limits
  (RLS enabled; browser never talks to DB with secret key)

7. BACKEND / DATABASE / SUPABASE WORK
================================================================================
Schema (supabase/schema.sql):
  waitlist
    id, email (unique), name, role, competition_interest, source,
    referral_code, referred_by, ip_hash, created_at
  api_rate_limits
    id, bucket_key, created_at
  Indexes on created_at, ip_hash, (bucket_key, created_at)
  Row Level Security enabled on both tables

API behaviors I implemented:
  • Server-only Supabase admin client (SUPABASE_SECRET_KEY never in browser)
  • Referral code generation (hash + salt)
  • Honest waitlist position: only shown when total signups ≥ 100
      (never uses row id as fake rank; never returns random fabricated ranks)
  • Duplicate email → idempotent success with existing referral code
  • Graceful retry if optional columns missing
  • "Other" role detail folded into competition_interest (no schema migration needed)

Ops I did outside the repo:
  • Created Supabase project
  • Ran SQL in Supabase SQL Editor
  • Added Vercel env vars (URL, publishable key, secret key, SITE_URL)
  • Verified live inserts in the waitlist table

8. SECURITY & ABUSE PROTECTIONS
================================================================================
Already shipping:
  • Origin / Referer allowlist (production blocks off-site POSTs)
  • IP-hashed rate limiting (8 requests / 10 minutes)
  • Honeypot fields (company / website) → silent fake success
  • Form timing check (too fast <1.2s or too stale >6h → fake success)
  • JSON-only + 4KB body size cap
  • Email regex + abuse heuristics
  • Role allowlist
  • Input sanitization + max lengths
  • Unique email constraint
  • Method guard (GET → 405)
  • Security headers: CSP, X-Frame-Options DENY, nosniff,
    Referrer-Policy, Permissions-Policy
  • RLS on tables; secret key server-only

Known non-blockers for a waitlist (documented, not required day one):
  • CAPTCHA only if spam appears
  • Confirm api_rate_limits exists in every environment (code fails open if missing)

12. PROBLEMS I DIAGNOSED & FIXED IN PRODUCTION
================================================================================
  • Missing Supabase env on Vercel → configured Production/Preview vars
  • 403 Invalid request → origin allowlist didn’t include Vercel production URL
      → extended getAllowedOrigins + NEXT_PUBLIC_SITE_URL
  • 429 / rate limit insert failures → api_rate_limits table missing
      → SQL + fail-open when table absent so signups aren’t blocked
  • Social link previews showing founder face → custom OG/Twitter images
      with Causey logo lockup
  • Waitlist rank showing DB id as “#2” → honesty gate + real count logic
  • Hero “See how it works” looking washed out → mask was on parent element
      (fixed by moving grid mask to ::before background layer)
  • Disabled CTA looking transparent → removed opacity: 0.72 on disabled state
